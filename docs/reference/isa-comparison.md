# BLIP ISA - cross-architecture syntax comparison

> **Non-normative.** A reference aid, not part of the justification chain (see [AGENTS.md](../AGENTS.md)). It names external architectures only for *syntax comparison*; nothing here justifies a BLIP decision.

Every BLIP opcode byte `0x00`-`0xFF` is listed; unused/reserved bytes show `-`. **Proposed** holds the agreed restyling of the BLIP cell (blank = no change); the rules are in the [Proposed](#proposed) section. Each used opcode is also translated into six other assembly languages, matched to that opcode's BLIP addressing mode.

**Legend:** `≈` = closest faithful multi-instruction equivalent; `—` = the ISA cannot express it directly. BLIP / 6502 / STM8 / Magic-1 use `$`-hex; 8080 / 8086 use Intel trailing-`h` hex. Generated from [isa.md](isa.md) §8; foreign columns adversarially cross-checked.

## Proposed

The **Proposed** column rewrites each BLIP cell by applying the rules below, cumulatively; a cell is left blank when no rule changes it. The goal is one consistent house style for BLIP assembly.

**R1 — Split the register off the verb.** Any operation that bakes its operand register into the mnemonic is written `VERB <reg>,…` (or just `VERB <reg>` when there is no operand). Covers loads, stores, 8-/16-bit ALU, the register read-modify-write forms, compares, and `LEA`.
  - `LDA #$05` → `LD A,$05`  ·  `SUBA 6,X` → `SUB A,(X+6)`  ·  `NEGB` → `NEG B`  ·  `LEAX 4,X` → `LEA X,X+4`
  - Mnemonics with **no** register designator keep their verb: the memory-operand RMW forms (`NEG $1234`, `CLR 6,X`) and the system ops (`DAA`, `SEX`, `ABX`, `ANDCC`, `LDMMU`, branches, `NOP`, …).

**R2 — Immediates as bare hex.** An immediate operand drops the `#` and is written as a `$`-prefixed hex literal. Applies to every instruction.
  - `SUBA #$05` → `SUB A,$05`  ·  `ANDCC #$EF` → `ANDCC $EF`  ·  `LDMMU #2` → `LDMMU $2`

**R3 — Memory is parenthesised; `(…)` means *dereference*.** Any operand that names a memory location is written in parentheses, and `(addr)` always means “the contents at `addr`”. A bare token is the value itself, so this also disambiguates immediates (`$05`) from memory (`($05)`). The address inside may be:
  - absolute — `($1234)`
  - a register, i.e. register-indirect (the register holds the address) — `(X)`, `(SP)`
  - register + displacement (Z80 order, register first) — `(X+6)`, `(SP-8)`
  - register + accumulator — `(X+B)`, `(X+D)`
  - with auto-increment / -decrement — `(X+)`, `(-X)`
  - a further level of indirection — `((X+6))`  ·  PC-relative — `(PC+6)`
  - `LDA 6,X` → `LD A,(X+6)`  ·  `LDA ,X` → `LD A,(X)`  ·  `STA $1234` → `ST A,($1234)`
  - **Bare = an address, not its contents.** Operands that *name an address* rather than dereference it stay unparenthesised: a `LEA` result and a jump/call target. `LEAX 4,X` → `LEA X,X+4`; `JMP ,X` → `JMP X`; `JMP $1234` → `JMP $1234`. (`JMP (X)` would instead jump *through* the pointer in X — BLIP's `JMP [,X]` → `JMP (X)`.)

**R4 — Register moves use `LD` / `XCHG`.** A register-to-register copy is spelled `LD dst,src`; a register exchange is `XCHG`. (The assembler emits the transfer/exchange opcode when both operands are registers — `LD` is overloaded over register, immediate, and memory operands, Z80-style.)
  - `TFR D,X` (D→X) → `LD X,D`  ·  `EXG A,B` → `XCHG A,B`
  - `LD` is `dst,src` where the old `TFR` was `src,dst`, so the operands **swap**; `XCHG` keeps its order (a swap is symmetric).
  - The privileged USP-banking forms become the USP variants: `TFRU` → `LD USP,X`, `EXGU` → `XCHG X,USP`.

Register-list operands (`PSHS A,B,X`, `PULS A,B,X`) and relative targets (`lbl`) are left unchanged.

**Decided.** Stores keep the R1 split form — `STA $1234` → `ST A,($1234)` — rather than the full Z80 `LD (…),A`, so BLIP keeps distinct `LD` / `ST` verbs (load vs. store stays visually explicit). Rules R1–R4 now cover the whole table; no operand kinds remain open.

## Addressing modes

This section compares how each architecture expresses the same **addressing modes** — the ways an instruction says *where its operand lives*. Every example loads the accumulator, so the operand notation is the only thing that varies. `≈` marks a different underlying mechanism or a multi-instruction stand-in; `—` marks a mode the architecture lacks.

**What each mode means:**

- **Immediate** — the operand is a constant baked into the instruction stream; no data memory is read to obtain the value. Shown in 8-bit and 16-bit widths.
- **Direct / extended (absolute)** — the instruction carries a full 16-bit address and accesses that fixed memory location.
- **Indexed, constant offset** — effective address = an index/base register plus a fixed signed displacement (`base + n`). The workhorse for C struct fields and stack locals (`n,SP`).
- **Register-indirect (zero offset)** — effective address = the index register itself (`base + 0`): a plain pointer dereference, `*p`.
- **Accumulator / register offset** — effective address = an index register plus the *value held in another register* (`base + A/B/D`): computes `array[i]` with no separate address arithmetic.
- **Auto-increment / -decrement** — dereference the pointer register, then step it (`*p++`), or step then dereference (`*--p`). Walks arrays and stacks with no extra instructions.
- **Indexed-indirect** — the computed address holds *another* address, which is then used: pointer-to-pointer and jump-table access, `**p`.
- **PC-relative (data)** — effective address = the program counter plus a displacement, so data references are position-independent (code relocatable without fixups).
- **Branch-relative (control)** — the branch target is the program counter plus a signed displacement, making control flow position-independent.
- **Inherent** — no operand at all; the instruction's effect is fully implied (e.g. `NOP`, `RTS`).
